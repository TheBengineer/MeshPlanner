# ITM Cross-Validation: JS Simplified Model vs Python Full Reference

**Date:** 2026-06-25
**Engine (Python):** Full `itmlogic` (qlrpfl/avar/lrprop subroutines)
**Engine (JS):** Simplified knife-edge diffraction (single dominant obstacle, ITU-R P.526)
**Profiles:** 10 canonical terrain types
**Frequency:** 915 MHz (US915 ISM band)

---

## 1. Overview

This document compares the path loss predictions of two implementations of the Irregular Terrain Model (ITM):

1. **Python reference** at `src/meshplanner/propagation/itm.py` -- the full Longley-Rice model using the `itmlogic` library. This is the authoritative engine that drives all production coverage calculations.
2. **JS simplified model** at `meshplanner-app/src/lib/propagation/itm.ts` -- a lightweight TypeScript port that uses single-knife-edge diffraction (ITU-R P.526) with a troposcatter floor, designed for the browser-based planning UI.

The 10 canonical profiles live in `tests/cross_validation/data/canonical/`. Each profile was generated by `generate_profiles.py`, which computed the Python golden values. The JS test harness (`test_js.test.ts`) loads the same profiles and produces the comparison data. The combined runner (`runner.py`) prints the terminal report used to produce the table below.

### 2.2 Free-Space Loss Agreement

Free-space path loss (FSPL) uses the same ITU-R P.525 formula in both engines:

```
FSPL(dB) = 32.45 + 20 * log10(f_MHz) + 20 * log10(d_km)
```

The JS and Python free-space values match exactly for all 10 profiles (deviation = 0.0 dB). This confirms the basic formula and distance/frequency inputs are consistent between implementations. All differences come from the excess loss component.

## 3. How the JS Model Works

The JS simplified model at `itm.ts` implements three steps:

1. **Free-space baseline**: Compute FSPL from distance and frequency (identical to Python).
2. **Single knife-edge obstruction**: Find the highest elevation point along the profile. If the terrain relief (max - min) exceeds 10 m and the path is longer than 1 km, compute the Fresnel-Kirchhoff diffraction parameter and apply ITU-R P.526 loss:
   ```
   A(v) = 6 + 20 * log10(v + sqrt(v^2 + 1))   for v > -0.7
   ```
3. **Troposcatter floor**: For paths longer than 10 km, cap excess loss at:
   ```
   A_tropo = 10 * log10(d_km) + 20
   ```

The model has no awareness of:
- Ground reflection / two-ray interference
- Atmospheric refraction (k-factor, effective Earth radius)
- Multiple obstructions (subsidiary ridges, terrain clutter)
- Climate zone or ground electrical parameters
- Polarization effects
- Surface wave propagation

## 4. Qualitative Assessment

### 4.1 Where the JS Model Performs Best

**Deep valley (delta = -4.5 dB)**: This profile descends from 200 m to 50 m then ascends again. The dominant terrain feature is a concave depression (valley), not a convex obstruction. Since the highest point is at one end of the profile (where the diffraction geometry collapses), the knife-edge calculation naturally returns near-free-space results -- which happens to align with the full ITM prediction (only 4.5 dB of excess loss). This is the only profile where the JS delta is under 10 dB.

**Hill and valley (delta = -19.0 dB)**: A single hill (120 m) followed by a valley produces the second-best result. The dominant obstacle (the hill) is relatively well-centered, so the knife-edge formula captures roughly 60% of the excess loss. The valley portion doesn't add much additional obstruction in the full ITM model, so the single-obstacle approximation holds moderately well.

### 4.2 Where the JS Model Performs Poorly

**Flat terrain (delta = -26.5 dB)**: The JS model returns pure free-space loss for flat terrain (terrain relief = 0, so the knife-edge block is skipped entirely). The full ITM adds 26.5 dB for ground wave propagation and surface roughness effects. This is a structural limitation: the knife-edge model has no concept of ground reflection or surface wave. A two-ray ground reflection model would be needed to capture this loss.

**Rolling hills (delta = -29.0 dB)**: With gentle undulations of 20-50 m amplitude but no single dominant peak, the knife-edge model only sees the highest point and misses the cumulative diffraction from multiple terrain undulations. The full ITM uses a statistical terrain roughness parameter that aggregates the effect of all hills.

**Double ridge (delta = -28.0 dB)**: Two distinct ridges (120 m and 200 m) create two sequential obstructions. The JS model only sees the higher of the two ridges (200 m) and ignores the shadowing effect of the first ridge. Full ITM accounts for multiple diffracting edges, which adds significant loss.

**Urban canyon (delta = -23.4 dB)**: The 5 km urban profile models buildings as 10-50 m terrain spikes. The knife-edge model sees the single highest "building" but misses the cumulative diffraction, reflection, and scattering from the many obstacles along the path.

### 4.3 Where the JS Model Fails Completely

**Asheville composite (delta = -41.9 dB)**: This realistic 25 km mountain profile simulates the terrain around Asheville, NC, with river valleys at 650 m rising to ridges at 1100 m+. The profile has multiple major ridges, rolling sections, and varying terrain. The JS model only diffracts over the single highest ridge (around 1118 m) and ignores all other terrain interaction. The full ITM integrates over the entire terrain profile, including multiple diffractions, terrain roughness, and atmospheric refraction. The 42 dB gap is roughly 25% of a typical LoRa link budget -- enough to render the prediction useless.

**Mountain ridge (delta = -30.5 dB)**: A single very tall ridge (350 m) at 915 MHz should create significant diffraction loss. The JS knife-edge formula captures about half the loss (33 dB vs 63.5 dB from full ITM). The difference likely comes from the ITM model's treatment of the ridge as a rounded/hilly obstruction rather than a perfect knife edge, plus terrain roughness on both sides of the ridge.

### 4.4 Systematic Bias

The JS model is **consistently optimistic** -- it always under-predicts path loss (every delta is negative). This means coverage predictions from the JS model will always suggest better coverage than the full ITM model would predict. In a disaster-response planning tool, this bias is dangerous: it could lead operators to believe a link will work when the signal is actually below threshold.

However, the bias is predictable. Roughly:

| Terrain complexity | Typical JS under-prediction |
|---|---|
| Flat / near-free-space | 25-30 dB |
| Single ridge or hill | 19-26 dB |
| Multiple ridges | 28-30 dB |
| Complex mountainous | 30-42 dB |

## 6. Recommendation

### 6.1 Should we proceed with the simplified JS model?

**Proceed for limited use, with documented caveats.**

The JS simplified model is acceptable for:

- **Quick visual previews** in the browser UI where the user needs a rough idea of coverage before running the full engine.
- **Flat to gently rolling terrain** (delta error 25-30 dB, but consistent enough to apply a band-level correction).
- **Single-obstruction paths** where the user knows the terrain type and can mentally account for the optimistic bias.

The JS model is **NOT acceptable** for:

- **Mountainous terrain** planning (the Asheville composite delta of 42 dB renders predictions meaningless).
- **Any path with multiple ridges** (the JS model always misses the second obstruction).
- **Production decision-making** where safety-of-life or critical communications depend on accurate coverage boundaries.
- **Optimization pipelines** (greedy site selection, ILP) where relative ranking of sites matters -- the JS model may rank sites incorrectly because it does not penalize multiple-obstruction paths enough.

### 6.2 Mitigations for the JS model

If the JS model is used in production before the WASM port is complete, apply these mitigations:

1. **Terrain-aware error correction**: Add a terrain-complexity heuristic to `computePathLoss()` that increases the excess loss based on the number of significant terrain peaks (e.g., count peaks above a threshold, multiply by a per-peak loss factor). This would reduce the double-ridge delta.
2. **Minimum excess loss floor**: For flat terrain, add a 20-25 dB floor to account for ground wave effects. Even a simple two-ray model would improve the flat-terrain result.
3. **Safety margin advisory**: Display a warning in the UI when terrain roughness exceeds a threshold (e.g., 300 m relief), advising the user that JS predictions may be optimistic by up to 40 dB.
4. **Profile-edge obstacle fix**: The knife-edge calculation collapses when the highest point is at index 0 or N-1 (as in the deep valley case). Check for this and fall back to free-space explicitly rather than producing a silent zero-excess result.

### 6.3 Should we trigger the full ITMLogic WASM port?

**Yes -- the WASM port is strongly recommended.**

The systematic errors in the JS model (20-42 dB across most terrain types) are too large for a production planning tool. A WASM port of the Python `itmlogic` library would:

- Eliminate the accuracy gap entirely (same numerical engine as the Python coverage module).
- Enable the browser UI to produce results identical to the Python backend.
- Handle all terrain types uniformly without per-type corrections.
- Remove the need to maintain two divergent propagation engines.

Estimated effort: 2-3 weeks for a competent engineer familiar with Emscripten or similar WASM toolchains. The `itmlogic` library is pure Python/numpy -- the core Fortran-derived subroutines (qlrpfl, avar, lrprop) would need to be ported to C or compiled via `wasm-pack`/`pyodide`.

### 6.4 Decision Matrix

| Criterion | JS Simplified Model | Full ITMLogic WASM |
|---|---|---|
| Accuracy vs Python reference | Poor to fair (5-42 dB error) | Identical (< 0.1 dB) |
| Implementation effort | Already done (80 lines) | 2-3 weeks |
| Bundle size | ~1 KB | ~200-500 KB (compiled) |
| Execution speed | < 1 ms per path | ~1-10 ms per path |
| All terrain types | Fails on complex terrain | Handles all ITM terrain types |
| Maintainability | Duplicated logic | Single engine, compiled |

Run the cross-validation at any time to regenerate the comparison:

```bash
# Full terminal report (Python + JS)
python3 tests/cross_validation/runner.py

# JSON output for automated parsing
python3 tests/cross_validation/runner.py --json

# Python-only unit tests
pytest tests/cross_validation/test_python.py -v

npx vitest run --project cross-validation
```

```bash
python3 tests/cross_validation/generate_profiles.py
```
